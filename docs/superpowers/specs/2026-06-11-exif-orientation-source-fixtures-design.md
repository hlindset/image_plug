# EXIF orientation source fixtures (2/3/4/5/7/8) ‚Äî design

Issue: [#204](https://github.com/hlindset/image_pipe/issues/204). Refs #203, #146, #185.

## Problem

The imgproxy differential conformance suite has exactly one EXIF source ‚Äî
`exif.jpg`, orientation **6** (90¬∞ CW quarter-turn). The other seven EXIF
orientations are untested. Several reach distinct `gravity.go RotateAndFlip` /
`orientation.ex` / `orientation_flush.ex` compose paths that orientation 6
(a pure quarter-turn, no flip) cannot. EXIF orientation lives in the **source
bytes** (the EXIF Orientation tag), not in a URL option, so each orientation
needs its own committed source fixture.

This change adds the six missing orientations (1 = identity is the no-op baseline
and needs no fixture) and a Batch-A-shaped constellation set on each.

EXIF orientation semantics: 1 = identity, 2 = flip-H, 3 = 180¬∞, 4 = flip-V,
5 = transpose (flip-H ‚àò 90¬∞), 6 = 90¬∞ CW *(have it)*, 7 = transverse
(flip-H ‚àò 270¬∞), 8 = 270¬∞ CW.

## Scope decisions (locked)

- **All six** missing orientations: 2, 3, 4, 5, 7, 8.
- **Full Batch-A mirror** ‚Äî ~22 new constellations (no trimming).
- **Test-data scoped PR.** If a transpose/transverse (5/7) case exposes a real
  ImagePipe storage‚Üîdisplay compose bug, it is **quarantined** with a new
  tracking issue; the fix lands in a follow-up PR, not here.

## Sources

### Base image ‚Äî reused unchanged

Every orientation fixture is the **identical base pixels** of today's `exif.jpg`,
differing only in the EXIF Orientation tag. The base is a 400√ó300 frame on a
`[200, 180, 60]` ground with a `[40, 40, 200]` blue block filling the top-left
quadrant `(0, 0, 200, 150)`. That single corner block in a non-square frame
distinguishes all eight orientations uniquely (corner position √ó frame aspect),
and is already proven by orientation 6. Reusing it keeps provenance review
trivial: *identical base, tag N*.

### Naming ‚Äî uniform `exif_N.jpg`

Rename the existing `exif.jpg` ‚Üí `exif_6.jpg` and add `exif_2.jpg`, `exif_3.jpg`,
`exif_4.jpg`, `exif_5.jpg`, `exif_7.jpg`, `exif_8.jpg`. Greenfield rename ‚Äî no
back-compat concern. The current `exif.jpg` provenance is regenerated by
`gen_sources`, so the manifest's `sources` key changes from `"exif.jpg"` to the
new filenames automatically.

### `gen_sources` change

In `test/support/mix/tasks/imgproxy.gen_sources.ex`, replace the single
orientation-6 `exif` build with a loop that builds one tagged file per
orientation in `[2, 3, 4, 5, 6, 7, 8]` from one shared base image:

```elixir
exif_base =
  400
  |> Image.new!(300, color: [200, 180, 60])
  |> Image.Draw.rect!(0, 0, 200, 150, color: [40, 40, 200])

for o <- [2, 3, 4, 5, 6, 7, 8] do
  exif_base
  |> Image.set_orientation!(o)
  |> write!("exif_#{o}.jpg", suffix: ".jpg", quality: 95)
end
```

(`set_orientation!` mutates the EXIF tag only; the stored pixels are identical
across all seven files.)

## Constellations

In `test/support/.../constellations.ex`:

- Replace the `@source_files` `exif_jpeg: "exif.jpg"` entry with
  `exif_2: "exif_2.jpg"` ‚Ä¶ `exif_8: "exif_8.jpg"` (atoms `:exif_2`‚Ä¶`:exif_8`).
- Re-point the two existing constellations (`rotate_exif`, `strip_exif`) from
  `:exif_jpeg` to `:exif_6` ‚Äî same source bytes, new atom/filename.

### Batch-A shape per orientation

Three base cases on each of the six new orientations, each hitting a distinct
seam (mirrors the issue's suggested set):

| opts | seam exercised |
|---|---|
| `rs:fill:200:150` | asymmetric cover ‚Äî per-axis storage‚Üîdisplay dims (landscape target vs portrait display on 5/7/8) |
| `c:200:120:no` | non-center crop gravity rotated into the storage frame |
| `rs:fit:200:200/ex:1:so` | extend with non-center (south) gravity applied **after** the orientation flush |

IDs: `exifN_cover`, `exifN_crop_no`, `exifN_extend_so` for each
`N ‚àà {2,3,4,5,7,8}` ‚Üí 18 base cases.

### 5/7 user-op crosses (deepest #146 compose)

For orientations **5 and 7 only**, two crosses each stacking an EXIF transpose
with a user op on one base geometry:

| id | opts |
|---|---|
| `exif5_cover_rot90` | `rs:fill:200:150/rot:90` |
| `exif5_cover_fl` | `rs:fill:200:150/fl:1` |
| `exif7_cover_rot90` | `rs:fill:200:150/rot:90` |
| `exif7_cover_fl` | `rs:fill:200:150/fl:1` |

‚Üí 4 crosses. **Total: 22 new constellations.**

All authored initially as `c(...)` ‚Äî `:equal` verdict, `:transform` group,
default tol (Œî2, budget 64).

**Grammar verification (hard gate before bake).** Each opt must parse in
ImagePipe *and* be accepted by the pinned imgproxy container, or `gen_fixtures`
produces no fixture for that case. The compatibility reviewer confirms against
the local imgproxy checkout:

- `c:200:120:no` ‚Äî inline crop-gravity form. If imgproxy/ImagePipe require the
  separated form, fall back to `c:200:120/g:no` (as the existing
  `crop_gravity_marker` uses `g:` separately).
- `rot:90` ‚Äî `rotate` option (already baked in #203, e.g. `rot:180`).
- `fl:1` ‚Äî user flip. The issue cites it as baked in #203 (T2.16/T2.18); the
  reviewer confirms the pinned imgproxy build accepts it. If it does **not**,
  drop the two `*_fl` crosses and replace with a second `rot` axis (`rot:270`),
  noting the substitution.

## Triage branch (post-bake)

After baking fixtures, each new case is one of:

1. **`:equal` as authored** ‚Äî keep.
2. **libvips-version AA/resampling skew** ‚Äî widen `tol` (raise budget, keep Œî2
   where possible) with the same evidence-based comment discipline as the
   existing cases: name the band-byte count, the seam location, max Œî, and the
   structural-flip argument that a real placement shift would blow the budget.
   No blanket threshold bumps.
3. **Real ImagePipe orientation compose bug** (`orientation.ex` /
   `pending_orientation.ex` / `orientation_flush.ex`) ‚Äî **quarantine** via a
   `triage:` key (reason + new tracking issue number) behind
   `--include imgproxy_triage`. Do **not** fix here. Open the follow-up issue and
   reference it.

## Regeneration

1. `mise exec -- mix imgproxy.gen_sources` ‚Äî rewrites the 6 `exif_N.jpg` sources.
2. `MIX_ENV=test IMGPROXY_DIFF=1 mise exec -- mix compile --force`
3. `MIX_ENV=test IMGPROXY_DIFF=1 mise exec -- mix imgproxy.gen_fixtures` ‚Äî pinned
   imgproxy Docker bakes PNG fixtures, rewrites `manifest.exs` + `REPORT.md`.
4. `MIX_ENV=test mise exec -- mix compile --force` ‚Äî rebuild without the gated
   generator module so a plain `mix test` doesn't see it.

Manifest provenance (source hashes, fixture hashes, libvips versions) is
regenerated, never hand-edited. `gen_sources` runs on the local (bleeding-edge)
libvips; the `imgproxy_libvips` vs ImagePipe-release provenance discipline from
#210 is unchanged.

## Testing & gate

- The conformance test auto-loops over `Constellations.all()`, so the 22 new
  cases are picked up with no test-file change.
- If (and only if) a real bug were fixed here, a request-boundary pixel test
  would accompany it ‚Äî but per scope, real bugs are quarantined, not fixed, so
  no production code or new unit test is expected.
- Close with `mise run precommit` (format, compile-warnings-as-errors, credo
  --strict, full `mix test`).

## Out of scope

- Fixing any orientation compose bug surfaced by 5/7 (‚Üí follow-up issue).
- Orientation 1 (identity) ‚Äî no observable transform, no fixture.
- New URL-option grammar ‚Äî all opts use existing parsed options.
