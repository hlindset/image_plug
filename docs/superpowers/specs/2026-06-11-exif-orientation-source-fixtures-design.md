# EXIF orientation source fixtures (2/3/4/5/7/8) — design

Issue: [#204](https://github.com/hlindset/image_pipe/issues/204). Refs #203, #146, #185.

## Problem

The imgproxy differential conformance suite has exactly one EXIF source —
`exif.jpg`, orientation **6** (90° CW quarter-turn). The other seven EXIF
orientations are untested. Several reach distinct `gravity.go RotateAndFlip` /
`orientation.ex` / `orientation_flush.ex` compose paths that orientation 6
(a pure quarter-turn, no flip) cannot. EXIF orientation lives in the **source
bytes** (the EXIF Orientation tag), not in a URL option, so each orientation
needs its own committed source fixture.

This change adds the six missing orientations (1 = identity is the no-op baseline
and needs no fixture) and a Batch-A-shaped constellation set on each.

EXIF orientation semantics: 1 = identity, 2 = flip-H, 3 = 180°, 4 = flip-V,
5 = transpose (flip-H ∘ 90°), 6 = 90° CW *(have it)*, 7 = transverse
(flip-H ∘ 270°), 8 = 270° CW.

## Scope decisions (locked)

- **All six** missing orientations: 2, 3, 4, 5, 7, 8.
- **Full Batch-A mirror** — ~22 new constellations (no trimming).
- **Test-data scoped PR.** If a transpose/transverse (5/7) case exposes a real
  ImagePipe storage↔display compose bug, it is **quarantined** with a new
  tracking issue; the fix lands in a follow-up PR, not here.

## Sources

### Base image — reused unchanged

Every orientation fixture is the **identical base pixels** of today's `exif.jpg`,
differing only in the EXIF Orientation tag. The base is a 400×300 frame on a
`[200, 180, 60]` ground with a `[40, 40, 200]` blue block filling the top-left
quadrant `(0, 0, 200, 150)`. That single corner block in a non-square frame
distinguishes all eight orientations uniquely (corner position × frame aspect),
and is already proven by orientation 6. Reusing it keeps provenance review
trivial: *identical base, tag N*.

### Naming — uniform `exif_N.jpg`

Rename the existing `exif.jpg` → `exif_6.jpg` and add `exif_2.jpg`, `exif_3.jpg`,
`exif_4.jpg`, `exif_5.jpg`, `exif_7.jpg`, `exif_8.jpg`. Greenfield rename — no
back-compat concern. The current `exif.jpg` provenance is regenerated by
`gen_sources`, so the manifest's `sources` key changes from `"exif.jpg"` to the
new filenames automatically.

### `gen_sources` change

In `test/support/mix/tasks/imgproxy.gen_sources.ex`, replace the single
orientation-6 `exif` build with a loop that builds one tagged file per
orientation in `[2, 3, 4, 5, 6, 7, 8]` from one shared base image:

```elixir
exif_base =
  400
  |> Image.new!(300, color: [200, 180, 60])
  |> Image.Draw.rect!(0, 0, 200, 150, color: [40, 40, 200])

for o <- [2, 3, 4, 5, 6, 7, 8] do
  exif_base
  |> Image.set_orientation!(o)
  |> write!("exif_#{o}.jpg", suffix: ".jpg", quality: 95)
end
```

(`set_orientation!` mutates the EXIF tag only; the stored pixels are identical
across all seven files.)

## Constellations

In `test/support/.../constellations.ex`:

- Replace the `@source_files` `exif_jpeg: "exif.jpg"` entry with
  `exif_2: "exif_2.jpg"` … `exif_8: "exif_8.jpg"` (atoms `:exif_2`…`:exif_8`).
- Re-point the two existing constellations (`rotate_exif`, `strip_exif`) from
  `:exif_jpeg` to `:exif_6` — same source bytes, new atom/filename.

The `@valid_sources` allowlist in
`test/image_pipe/imgproxy_differential/constellations_test.exs` (which asserts
every constellation's source is a known atom) must drop `:exif_jpeg` and gain
`:exif_2`…`:exif_8` in the same change, or that well-formedness test fails.

### Batch-A shape per orientation

Three base cases on each of the six new orientations, each hitting a distinct
seam (mirrors the issue's suggested set):

| opts | seam exercised |
|---|---|
| `rs:fill:200:150` | asymmetric cover — per-axis storage↔display dims (landscape target vs portrait display on 5/7/8) |
| `c:200:120/g:no` | non-center crop gravity rotated into the storage frame |
| `rs:fit:200:300/ex:1:so` | extend with non-center (south) gravity applied **after** the orientation flush |

IDs: `exif_N_cover`, `exif_N_crop_no`, `exif_N_extend_so` for each
`N ∈ {2,3,4,5,7,8}` → 18 base cases.

The extend target is `200:300` (portrait), **not** square `200:200`. A square
target leaves the south-gravity extend with no vertical play on the
portrait-display quarter-turns (5/6/7/8 fit height-bound, already filling the
canvas height), so it would not exercise the post-flush gravity seam exactly
where it matters most. `200:300` keeps real vertical play for both the
landscape (2/3/4) and portrait (5/7) orientation classes. The crop uses the
separated `c:W:H/g:no` form (proven by the existing passing `crop_gravity_marker`
case), not the inline `c:W:H:no` colon form.

### 5/7 user-op crosses (deepest #146 compose)

For orientations **5 and 7 only**, two crosses each stacking an EXIF transpose
with a user op on one base geometry:

| id | opts |
|---|---|
| `exif_5_cover_rot90` | `rs:fill:200:150/rot:90` |
| `exif_5_cover_fl` | `rs:fill:200:150/fl:1` |
| `exif_7_cover_rot90` | `rs:fill:200:150/rot:90` |
| `exif_7_cover_fl` | `rs:fill:200:150/fl:1` |

→ 4 crosses. **Total: 22 new constellations.**

All authored initially as `c(...)` — `:equal` verdict, `:transform` group,
default tol (Δ2, budget 64).

**Grammar — confirmed.** A parallel compatibility review against the local
imgproxy v4.x source (`/Users/hlindset/src/imgproxy`) confirmed every opt is
accepted by the OSS `darthsim/imgproxy` build: crop `c:W:H/g:gravity`, gravity
tokens `no`/`so`, extend `ex:1:gravity`, `rot` (multiple-of-90 enforced), and
`fl` (flip is OSS, **not** Pro-only — `processing/rotate_and_flip.go`, no build
tag). All five shapes are also already exercised in the existing suite / #203 on
the ImagePipe side. A cheap pre-bake parse check (build a `%Plug.Conn{}`, call
`ImagePipe.Parser.Imgproxy.parse/1`) guards against a typo before the Docker
bake — but no grammar fallback is expected.

## Triage branch (post-bake)

After baking fixtures, each new case is one of:

1. **`:equal` as authored** — keep.
2. **libvips-version AA/resampling skew** — widen `tol` (raise budget, keep Δ2
   where possible) with the same evidence-based comment discipline as the
   existing cases: name the band-byte count, the seam location, max Δ, and the
   structural-flip argument that a real placement shift would blow the budget.
   No blanket threshold bumps.
3. **Real ImagePipe orientation compose bug** (`orientation.ex` /
   `pending_orientation.ex` / `orientation_flush.ex`) — **quarantine** via a
   `triage:` key (reason + new tracking issue number) behind
   `--include imgproxy_triage`. Do **not** fix here. Open the follow-up issue and
   reference it.

## Regeneration

1. `mise exec -- mix imgproxy.gen_sources` — rewrites the 6 `exif_N.jpg` sources.
2. `MIX_ENV=test IMGPROXY_DIFF=1 mise exec -- mix compile --force`
3. `MIX_ENV=test IMGPROXY_DIFF=1 mise exec -- mix imgproxy.gen_fixtures` — pinned
   imgproxy Docker bakes PNG fixtures, rewrites `manifest.exs` + `REPORT.md`.
4. `MIX_ENV=test mise exec -- mix compile --force` — rebuild without the gated
   generator module so a plain `mix test` doesn't see it.

Manifest provenance (source hashes, fixture hashes, libvips versions) is
regenerated, never hand-edited. `gen_sources` runs on the local (bleeding-edge)
libvips; the `imgproxy_libvips` vs ImagePipe-release provenance discipline from
#210 is unchanged.

## Testing & gate

- The conformance test auto-loops over `Constellations.all()`, so the 22 new
  cases are picked up with no test-file change.
- If (and only if) a real bug were fixed here, a request-boundary pixel test
  would accompany it — but per scope, real bugs are quarantined, not fixed, so
  no production code or new unit test is expected.
- Close with `mise run precommit` (format, compile-warnings-as-errors, credo
  --strict, full `mix test`).

## Out of scope

- Fixing any orientation compose bug surfaced by 5/7 (→ follow-up issue).
- Orientation 1 (identity) — no observable transform, no fixture.
- New URL-option grammar — all opts use existing parsed options.
